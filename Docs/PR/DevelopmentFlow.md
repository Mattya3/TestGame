# チーム開発の流れ（Issue 駆動開発・Git/GitHub ワークフロー）

本プロジェクトでは，作業を Issue 単位で分け，**必ず Pull Request（PR）経由でレビューしてからマージ**します．
目的は，作業の見通しを良くし，レビューしやすくし，壊れた変更を `main` に入れないことです．

## 1. 基本方針

- `main` へ直接 push しません．
- 1 つの PR では，**1 つの目的**に絞って変更します．
- 作業開始前に，必ず対応する Issue を確認します．
- レビュー前に，作成者自身が最低限の動作確認を行います．

## 2. Issue の種類とブランチ構造

### 🍎 main issue

- **目的**: 大きな機能単位を表します．
- **ブランチ名**: `feature/機能名/#main_issue番号`
- **マージ先**: `main`
- **役割**:
  - 機能全体の親ブランチです．
  - 複数の `sub issue` を束ねます．

### 🍏 sub issue

- **目的**: 実装・レビュー・マージできる最小単位の作業です．
- **ブランチ名**: `subfeature/機能名/#sub_issue番号`
- **マージ先**: 対応する `feature/〜` ブランチ
- **役割**:
  - 原則として，日常的な実装はこの単位で進めます．
  - 1 PR 1 目的の原則を守りやすくなります．

### 🐛 debug issue

- **目的**: バグ修正や不具合対応です．
- **ブランチ名**: `debug/修正内容/#debug_issue番号`
- **マージ先**: `main` または対応する `feature/〜` ブランチ
- **役割**:
  - 単発の不具合修正に使います．
  - 機能追加と混ぜず，不具合修正として切り出します．

## 3. 作業開始からマージまでの流れ

### Step 1. Issue を作る・引き受ける

1. まず GitHub Issues 上で，対応する `main issue` / `sub issue` / `debug issue` を確認または作成します．
2. 自分が担当する Issue にアサインします．
3. 完了条件が曖昧なら，実装前に Issue 本文を更新して明確にします．

### Step 2. ブランチを切る

1. `main issue` の担当者は，必要なら親ブランチ `feature/〜` を作成します．
2. 各担当者は，親ブランチから `subfeature/〜` あるいは `debug/〜` ブランチを切ります．
3. ブランチ名には，**内容と Issue 番号が分かる名前**を付けます．

### Step 3. 実装する

1. 変更は小さく区切ってコミットします．
2. Unity の Scene / Prefab / `.meta` 差分は，意図したものだけが入っているかこまめに確認します．
3. 作業中に別件を見つけた場合は，原則として別 Issue / 別 PR に分けます．

### Step 4. 自分で確認する

PR を出す前に，少なくとも以下を確認します．

- 対応 Issue の完了条件を満たしているか
- コンパイルエラーが出ていないか
- 主要な操作で期待通りに動くか
- `Debug.Log` や検証コードが残っていないか
- 無関係なファイル差分が混ざっていないか
- Scene / Prefab / `.meta` の組が崩れていないか

### Step 5. PR を作成する

1. 作業ブランチを push します．
2. GitHub で PR を作成します．
3. **マージ先が正しいかを必ず確認**します．
   - `subfeature/〜` は親の `feature/〜` へ
   - `feature/〜` は `main` へ
   - `debug/〜` は修正対象に応じたブランチへ
4. PR テンプレートに従って，少なくとも以下を埋めます．
   - 何を変えたか
   - なぜ変えたか
   - どう確認すればよいか
   - レビュアーに特に見てほしい点
5. レビュアーを 1 人以上指定します．

> テンプレートは [sub_issue.md](../../.github/PULL_REQUEST_TEMPLATE/sub_issue.md)，[main_issue.md](../../.github/PULL_REQUEST_TEMPLATE/main_issue.md)，[pull_request_template.md](../../.github/pull_request_template.md) を参照してください．

### Step 6. 自動整形の反映を取り込む

- PR 作成後や更新後に，GitHub Actions による自動整形コミットが入ることがあります．
- その後も同じブランチで作業を続ける場合は，**必ず先に `git pull` して整形差分を取り込んでから**再開してください．

### Step 7. レビュー対応をする

1. レビューコメントを確認します．
2. 指摘に対応したら，同じブランチへ追加コミットして push します．
3. 直さない判断をする場合は，理由を PR 上で説明します．
4. 必要なら再レビューを依頼します．

### Step 8. マージする

1. 必要な承認が揃い，指摘対応が完了したらマージします．
2. マージ後，作業ブランチは削除します．
3. `sub issue` をマージしたら，Issue / PR の状態を整理します．

## 4. PR を出す前に避けるべきこと

- 大量の unrelated な変更をまとめて出す
- 動作確認なしで PR を出す
- 何を確認すればよいか分からない PR 説明のまま出す
- テスト用オブジェクトや個人用 Scene 変更を混ぜたまま出す
- 「あとで直す前提」の壊れた状態でレビュー依頼する

## 5. 大きな機能の進め方

- 大きな機能は，まず `main issue` で全体像を定義します．
- 実装はなるべく `sub issue` に分割し，**小さな PR を積み上げて完成させる**形を取ります．
- 最後に `feature/〜` から `main` へ向けて統合 PR を作成します．
- 統合 PR では，個別 PR で見た内容の再確認に加えて，**統合時の不整合や動作崩れ**を重点的に確認します．
