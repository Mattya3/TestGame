# レビューのプロセスと方法

このドキュメントは，PR をレビューする側の基準をまとめたものです．
目的は，**壊れた変更を止めること**，**変更の意図を揃えること**，**チーム全体の実装品質を上げること**です．

## 1. レビューの基本姿勢

- レビューは，まず **正しさ** を見る作業です．
- 可読性や規約違反の確認は重要ですが，**バグ・回帰・エラー処理不足より優先しません**．
- コメントはコードに向けて行い，人に向けて攻撃しません．
- 指摘はできるだけ具体的に書きます．
- 問題がなければ，その判断にも責任を持って Approve します．

## 2. レビューで最初に見る順番

以下の順番で確認すると，重要な問題を見落としにくくなります．

1. **要件を満たしているか**
2. **コードや設定が単純に間違っていないか**
3. **壊れやすい箇所や回帰リスクがないか**
4. **エラー処理や異常系への備えがあるか**
5. **Unity 特有の事故を起こしていないか**
6. **可読性・保守性・規約の問題がないか**

## 3. 着眼点

### 3.1 正しさ

- Issue の完了条件を満たしているか
- 条件分岐や状態遷移が正しいか
- 既存仕様を壊していないか
- 端のケースで破綻しないか

### 3.2 エラー処理・安全性

- null 参照の危険がないか
- 設定漏れ時に分かる失敗になっているか
- 失敗時に不正な状態を残さないか
- 例外やエラーが黙殺されていないか

### 3.3 Unity 特有の観点

- `Awake` / `Start` / `OnEnable` / `OnDisable` の使い分けが適切か
- `Update()` / `FixedUpdate()` に重い処理が入っていないか
- `GetComponent` や `Find` 系を毎フレーム呼んでいないか
- `Scene` / `Prefab` / `.meta` の差分が危険でないか
- `Missing Script` や参照切れの可能性がないか

### 3.4 可読性・保守性

- 名前だけで役割が追えるか
- 1 クラス 1 責務に近いか
- public API が必要以上に広くないか
- コメントが「なぜ」を補えているか

## 4. コメントの分類

レビューコメントの先頭には，原則として以下のプレフィックスを付けます．

| プレフィックス | 意味 |
|---|---|
| `[MUST]` | 承認前に直してほしい問題．バグ，回帰リスク，危険な実装，重大なルール違反 |
| `[SHOULD]` | 強く直したい改善点．保守性や安全性を明確に上げる変更 |
| `[COULD]` | 任意の改善案．直すと良いが必須ではない |
| `[WANT]` | 好み寄りの提案．無視してよい |

### プレフィックスの使い分け

- **間違っている・壊れる・危険**なら `[MUST]`
- **今直す価値が高い改善**なら `[SHOULD]`
- **任意の改善案**なら `[COULD]`
- **完全に好み**なら `[WANT]`

## 5. コメントの書き方

レビューコメントには，可能な限り以下を含めます．

1. **依頼**: 何を直してほしいか
2. **理由**: なぜ必要か
3. **影響**: 放置すると何が起こるか，または直すと何が良くなるか

### 例

> **[MUST]** この null チェックを追加してもらえませんか？  
> `PlayersManager` の参照が未設定でも `Start()` まで進んでしまい，実行時に例外で止まる可能性があります．  
> ここで早めに検知できると，設定ミスをレビュー段階でも追いやすくなります．

## 6. レビュー時に避けること

- 根拠のない好みだけで `[MUST]` を付ける
- 自分でも説明できない違和感だけを投げる
- 何を確認したか分からないまま Approve する
- 小さな表現差だけを大量に指摘して，重要な問題を埋もれさせる

## 7. 動作確認

- C# のロジック変更，Scene 変更，Prefab 変更を含む PR は，可能なら手元で確認します．
- 小さな文書修正や明らかな軽微差分は，コードレビューのみでも構いません．
- 手元確認した場合は，PR 上に「何を確認したか」を残すと親切です．

## 8. Copilot レビューの使い方

- Copilot レビューは，見落としを減らすための補助として使って構いません．
- ただし，**Copilot のコメントだけで Approve の根拠にしない**でください．
- Copilot に指摘がなくても，人間レビューで正しさ・仕様適合・Unity の実機挙動を確認してください．
- Copilot の指摘を採用する場合も，妥当性を自分で判断してください．

## 9. レビュー完了からマージまで

- 指摘が残っている間は，Approve しません．
- 作成者が修正したら，差分だけでなく **指摘が解消されたか** を再確認します．
- 問題が解消し，十分に確認できたら Approve します．
- マージ後に気づいた問題は，責めるより先に再現条件と影響範囲を整理して共有します．
