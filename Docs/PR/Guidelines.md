# Pull Request（PR）とコミットの作法

チーム開発でレビューを機能させるには，PR の出し方そのものに品質が必要です．
このドキュメントでは，**レビューしやすく，安全にマージできる PR** を作るための基準をまとめます．

## 1. コミットの粒度

- コミットは，**意味のある小さい単位**で切ってください．
- 1 コミットで複数の無関係な目的を混ぜないでください．
- 「1 日分まとめて 1 コミット」は避けます．
- 失敗時に戻しやすく，差分を追いやすい粒度を目指します．

### 良い粒度の例

- 1 つの関数の追加
- 1 つのバグ修正
- 1 つの Prefab 調整
- 1 つのリファクタとそれに付随する安全な差分

### 避けたい粒度の例

- 機能追加，バグ修正，リネーム，ドキュメント更新を一度に入れる
- Scene の大量変更とコード変更を雑にまとめる
- 意図が読み取れない細切れコミットを大量に積む

## 2. コミットメッセージ

- 先頭にプレフィックスを付けます．
- 何をしたかが一読で分かる文にします．
- `修正`, `変更`, `wip`, `a` のような曖昧なメッセージは避けます．

| プレフィックス | 用途 | 例 |
|---|---|---|
| `feat:` | 新機能追加 | `feat: プレイヤーのジャンプ判定を追加` |
| `fix:` | バグ修正 | `fix: ゴール時に二重遷移する不具合を修正` |
| `refactor:` | 振る舞いを変えない整理 | `refactor: カメラ追従処理を分割` |
| `docs:` | ドキュメント更新 | `docs: レビュー手順を更新` |
| `chore:` | 環境・設定変更 | `chore: フォーマッタ設定を更新` |
| `clean:` | 不要コード削除 | `clean: 未使用のログ出力を削除` |
| `rename:` | 名前変更 | `rename: PlayerMove を PlayerMovement に変更` |
| `move:` | ファイル移動 | `move: UI 関連スクリプトを UI フォルダへ移動` |
| `delete:` | 削除 | `delete: 廃止したテスト Scene を削除` |

## 3. PR タイトル

- PR タイトルも，基本的にコミットと同じプレフィックスを使います．
- タイトルだけで変更の主目的が分かるようにします．
- 1 PR に複数目的が入っているなら，まず PR の切り方を見直してください．

## 4. PR 説明文に必ず書くこと

PR 本文には，少なくとも以下を書いてください．

1. **変更内容**: 何を変えたか
2. **背景**: なぜ必要か
3. **確認方法**: レビュアーがどう確認すればよいか
4. **確認結果**: 作成者が何を確認したか
5. **補足**: 未解決事項，見てほしい点，既知の制約

### 良い説明の例

- 変更内容が箇条書きで整理されている
- Scene 名や操作手順が具体的に書かれている
- 「未確認のケース」があるなら明記されている

### 不十分な説明の例

- 「いろいろ修正しました」
- 「動きます」
- 手元確認の方法が書かれていない

## 5. PR の大きさ

- 1 PR 1 目的を守ります．
- 無関係な修正は別 PR に分けます．
- 変更量はできるだけ小さく保ちます．

### 目安

- コード変更は **500 行以内** を推奨
- 変更ファイル数は **20 ファイル未満** を推奨

この基準を超えること自体が即 NG ではありませんが，超える場合は以下のどちらかを検討してください．

- 分割できないか見直す
- 分割できない理由を PR に書く

## 6. PR 作成前セルフチェック

### 正しさ

- [ ] Issue の完了条件を満たしている
- [ ] 主要な操作で期待した結果になる
- [ ] 既存機能を壊していない
- [ ] エッジケースや異常系を最低限考慮している

### エラー対処

- [ ] null 参照や設定漏れで落ちやすい箇所を確認した
- [ ] エラー時に黙って壊れない実装になっている
- [ ] 必要なガード節，検証，ログが入っている

### 差分の健全性

- [ ] 無関係な変更が混ざっていない
- [ ] `Debug.Log` や検証コードが残っていない
- [ ] コメントアウトした古いコードを残していない
- [ ] `Scene` / `Prefab` / `.meta` の差分が意図通り

### 可読性・保守性

- [ ] 名前から役割が分かる
- [ ] `MonoBehaviour` に責務を詰め込みすぎていない
- [ ] `Update()` / `FixedUpdate()` に重い処理を書いていない
- [ ] 他人が追える PR 説明になっている

## 7. Unity プロジェクト特有の確認項目

- `.meta` ファイルの付け忘れがないか
- `Missing Script` を生む変更になっていないか
- Scene 上の一時オブジェクトや個人検証用変更が残っていないか
- 再利用するものを Scene 直置きだけで済ませていないか
- レビュアーが確認できる Scene / Prefab / 操作手順が書かれているか

## 8. レビュー依頼の出し方

- 何を見てほしいかを PR 本文の「備考」やコメントで明示します．
- 自信がない点は隠さず書きます．
- 「ここは特に壊れやすい」「ここは設計相談もしたい」のように，レビューの観点を渡してください．

## 9. Copilot レビューの扱い

- Copilot レビューは，**人間レビューの補助**として使います．
- Copilot の指摘だけで安全と判断しないでください．
- Copilot が何も言っていなくても，人間が確認すべき項目は残ります．
- 特に Unity の Scene / Prefab の意図やゲーム仕様は，人間が文脈を補って判断する必要があります．
